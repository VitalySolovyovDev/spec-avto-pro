# План рефакторинга

## Цель

Привести проект к состоянию, в котором он:

- легко поддерживается;
- не содержит временных решений в публичных пользовательских сценариях;
- имеет понятную структуру исходников;
- готов к дальнейшему развитию без повторной пересборки архитектуры.

## Этап 1. Довести до публикационного минимума

1. Принять решение по карте: подключить реальную карту или убрать placeholder-блок.
2. Утвердить реальные внешние ссылки компании: мессенджеры, соцсети, дополнительные каналы связи.

## Этап 2. Привести форму к реальному рабочему сценарию

1. Выбрать канал обработки заявок:
   - backend API;
   - CRM webhook;
   - email service;
   - Telegram / internal notifier.
2. Добавить клиентскую валидацию телефона и обязательных полей.
3. Реализовать обработку состояний формы:
   - отправка;
   - успешная отправка;
   - ошибка.
4. Убрать зависимость от `mailto:` как основного пути отправки.
5. Добавить простую аналитику по отправке формы.

## Этап 3. Упростить структуру HTML

1. Разбить `src/index.html` на логические блоки:
   - header;
   - hero;
   - about;
   - services;
   - equipment;
   - process;
   - legal;
   - reviews;
   - faq;
   - contact;
   - footer.
2. Вынести повторяющиеся карточки в data-массивы:
   - преимущества;
   - виды отходов;
   - техника;
   - этапы;
   - отзывы;
   - FAQ.
3. Выбрать подход рендера:
   - build-time шаблоны;
   - простой JS-render из data-модулей;
   - lightweight component layer.
4. Вынести SVG sprite в отдельный локальный файл `src/assets/icons/`.

## Этап 4. Упростить и нормализовать CSS

1. Сохранить `variables.css` как слой токенов.
2. Оставить `base.css` для reset и базовых правил.
3. Разбить стили лендинга на более узкие файлы:
   - `layout.css`
   - `hero.css`
   - `cards.css`
   - `form.css`
   - `footer.css`
4. Проверить и удалить дублирующиеся и неиспользуемые правила после разбиения.
5. Ввести единое правило именования для секций и компонентов.

## Этап 5. Оптимизировать ассеты

1. Сжать текущие JPG-изображения без заметной потери качества.
2. Подготовить `webp` или `avif`-версии основных изображений.
3. Добавить responsive image strategy для hero и review-изображений.
4. Прелоадить ключевой шрифт или критический font-face, если это даст выигрыш.
5. Проверить итоговую сборку по warning-ам `Rspack` и метрикам Lighthouse.

## Этап 6. Добавить инженерные проверки

1. Подключить `ESLint` для JavaScript.
2. Подключить `Prettier` для единого форматирования.
3. Добавить smoke-проверку:
   - сайт собирается;
   - главная страница открывается;
   - страница политики открывается;
   - ключевые ссылки существуют.
4. При необходимости добавить простой browser-based сценарий:
   - раскрытие мобильного меню;
   - раскрытие FAQ;
   - прохождение формы.

## Этап 7. Финальная нормализация проекта

1. Обновить `README.md` после завершения рефакторинга.
2. Синхронизировать `.docs/architecture.md` с итоговой структурой.
3. Зафиксировать соглашения по контенту, ассетам и стилям.
4. Определить, что считается generated-файлами, а что исходниками.
5. При необходимости добавить шаблон релизного чек-листа.

## Быстрый practical roadmap

### В ближайший день

1. Закрыть политику конфиденциальности.
2. Решить судьбу карты и лицензии.
3. Сжать тяжелые изображения.
4. Убрать `mailto:` и подключить рабочую отправку формы.

### В ближайшую неделю

1. Вынести контент секций в данные.
2. Разбить `index.html` на более управляемые блоки.
3. Разделить CSS лендинга на мелкие доменные файлы.
4. Добавить lint и smoke-проверки.

### После стабилизации

1. Подключить аналитику и отслеживание лидов.
2. Добавить реальные внешние интеграции.
3. Подготовить проект к полноценной поддержке и публикации.
